# DataScope Studio 产品设计

更新时间：2026-07-06  
当前版本基线：`0.3.1`

## 1. 产品定位

DataScope Studio 是一个本地优先的数据可视化与快速数据诊断工作台。它面向机器人、工业设备、传感器、计算机视觉和实验数据，把不同来源的数据导入到统一 workspace，通过自动识别、Schema 检查、语义 Mapping、Rerun 可视化、查询索引和诊断报告，帮助用户快速回答三个问题：

1. 这份数据能不能被正确读取和可视化？
2. 这份数据里是否存在明显质量问题或故障信号？
3. 多次运行、多个数据源之间有什么差异，能否复现和共享？

建议的产品心智不只是“各种图表工具”，而是“数据体检 + 可视化回放 + 可复用诊断流程”。可视化负责让用户看见数据，诊断负责让用户更快知道哪里值得看。

## 2. 目标用户

| 用户 | 核心诉求 | 当前匹配度 |
| --- | --- | --- |
| 机器人算法与系统工程师 | 快速查看 ROS2/MCAP bag、定位 topic 缺失、时间同步、低电量、日志错误等问题 | 高 |
| 工业设备与传感器工程师 | 导入 CSV/JSONL/TXT 日志，查看指标趋势、状态变化、异常样本 | 中高 |
| 计算机视觉工程师 | 查看图片目录、检测框、预测置信度和失败帧 | 中高 |
| 数据工程/测试工程师 | 批量导入、校验字段、导出查询结果、生成可复现项目包 | 中 |
| 非代码用户 | 希望不用写脚本也能完成数据预览、问题定位和报告导出 | 中，需要继续简化 UI |

## 3. 核心场景

### 3.1 快速导入并可视化

用户选择本地文件或目录，系统自动识别数据类型，生成 schema profile 和 Mapping 草稿，用户确认后生成 `.rrd` Recording 与 `.rbl` Blueprint，并在 Rerun 中查看。

### 3.2 数据健康体检

用户对一个项目或若干 recording 运行诊断，系统输出健康分、严重级别、问题列表、证据、建议和可导出的 JSON/CSV/HTML 报告。

### 3.3 实验对比

用户选择多次运行，按 battery、state、scalar 等指标对比摘要或序列，判断不同实验、模型、参数或设备状态的差异。

### 3.4 模板复用

用户把确认过的 Mapping 保存为模板，在同类数据源上复用，并通过 diff 检查字段匹配变化，减少重复配置。

### 3.5 离线迁移与共享

用户把项目元数据、mapping、recording、blueprint、导出结果打包为 `.datascope.zip`，在其他机器上重新打开。

## 4. 当下已有能力

### 4.1 数据接入

当前已经内置多类 adapter：

| 类型 | 支持数据 |
| --- | --- |
| 表格数据 | CSV、JSONL/NDJSON |
| 文本表格与日志 | TSV、TXT、LOG、DAT、LST、LIST |
| 计算机视觉 | 图片文件或图片目录，支持 `annotations.json`、`predictions.json` sidecar |
| 点云 | PLY、PCD、NPY、NPZ、XYZ、XYZN、XYZRGB、PTS、ASC 文件或目录 |
| 机器人数据 | MCAP、ROS2 DB3 bag 或 bag 目录 |

已支持复制到项目和引用原路径两种存储方式，适合自包含项目和大型本地数据两种场景。

### 4.2 Workspace 与 Catalog

当前项目已经具备本地 workspace 和 SQLite catalog：

- 项目、数据源、stream、mapping、recording、任务、标签、导出结果统一登记。
- 默认 workspace 位于 `~/.datascope-studio`。
- 项目目录包含 `raw/`、`cache/`、`recordings/`、`blueprints/`、`mappings/`、`logs/`、`exports/`。
- 支持项目导入导出，便于备份、迁移和复现。

### 4.3 Schema Inspector 与自动 Mapping

当前已有的 schema profile 能检查：

- 字段名、字段类型、空值数量和空值比例。
- 时间字段是否存在、是否可解析、是否单调、推断时间单位。
- 图像、点云、MCAP、ROS2 DB3 等非表格源的 adapter metadata。

Mapping 侧已经支持：

- 自动推荐 stream 和 semantic type。
- 选择可视化模板，如 `sensor_monitor`、`cv_detection`、`robotics_debug`、`experiment_compare`。
- 校验 Mapping，并区分 error 与 warning。
- 保存、导入、导出和应用 Mapping 模板。
- 对同一个模板在两个数据源上的应用结果做 diff。

### 4.4 Rerun 可视化输出

当前已经能够生成：

- `.rrd`：Rerun recording。
- `.rbl`：Rerun blueprint。

支持从桌面端、API、CLI 打开 Rerun viewer。构建结果会记录 artifact 信息，包括文件大小、模板、converter、Rerun 版本、MCAP decoder、校验模式、Catalog 注册状态等。

### 4.5 查询、对比与导出

当前转换阶段会生成轻量 `query_rows` 索引，后续查询优先读取 SQLite，不需要重新解析原始文件或 `.rrd`。

内置查询模板：

| 模板 | 用途 |
| --- | --- |
| `find_errors` | 查找日志和状态中的 ERROR/WARN/fault/error |
| `low_battery` | 查找低电量样本 |
| `detection_failure` | 查找低置信度或无预测框的检测帧 |
| `topic_summary` | 汇总 MCAP/ROS2 topic 信息 |
| `state_duration` | 估算状态持续时间 |
| `time_sync` | 比较 topic 或传感器时间范围 |

查询结果支持 CSV 导出，安装 `pyarrow` 后支持 Parquet。多 recording 对比已经支持 summary 和 series 两类模式。

### 4.6 离线诊断

当前已有机器人诊断报告能力：

- topic coverage。
- ROS2 可转换性。
- message volume。
- time sync。
- logs and states。
- battery。
- CV detection。

报告包含健康分、严重级别、检查项、findings、证据和建议，支持 JSON、CSV、HTML 导出。

### 4.7 桌面端、API 与 CLI

当前已有三类入口：

- Tauri + React 桌面端：Dashboard、Import、Recordings、Diagnostics、Templates、Settings。
- FastAPI 本地服务：项目、导入、Mapping、构建、查询、诊断、插件、模板、批量导入、项目包等接口。
- Typer CLI：适合脚本化 inspect、import、query、diagnose、project export/import。

### 4.8 性能和稳定性基础

当前已有的工程基础包括：

- API 启动异步预热 workspace。
- 导入 workflow 合并多步请求，减少 UI 往返。
- 转换任务异步队列、取消、重试、进度写库节流。
- 查询按模板过滤，并对常见模板使用 cursor 和 limit。
- 大列表在前端限制渲染数量。
- 构建前有磁盘空间估算和 artifact 冲突处理。

## 5. 当前不足与机会

| 方向 | 当前不足 | 产品机会 |
| --- | --- | --- |
| 内置可视化 | 主要依赖 Rerun，桌面端内联图表较弱 | 在导入和诊断阶段加入轻量趋势图、分布图、缺失矩阵、图片网格和点云预览 |
| 通用数据诊断 | 机器人/CV 诊断较强，通用表格质量诊断还不完整 | 增加缺失、异常值、重复、常量列、分布漂移、时间间隔、单位混用等检查 |
| 查询体验 | 查询模板固定，参数能力有限 | 建设无代码查询构建器和用户自定义规则 |
| 诊断到可视化联动 | findings 目前更像报告项 | 每个 finding 应能跳转到对应 recording、entity path、时间点和相关样本 |
| 模板体系 | 已有模板骨架，但还偏工程能力 | 抽象为“诊断配方”：adapter + mapping + query + 阈值 + blueprint + report layout |
| 协作共享 | 已有项目包，但团队协作和版本比较较弱 | 增加项目版本、报告快照、模板版本和审阅备注 |
| 新手体验 | Mapping 能力强，但术语偏专业 | 增加字段推荐解释、修复建议、风险提示和一键接受安全修改 |

## 6. 建议的产品形态

### 6.1 从“可视化工具”升级为“数据诊断工作台”

建议把产品首页和核心文案聚焦为：

> 导入任意本地数据，自动生成可视化 Recording，并给出可复现的数据健康报告。

这样比“各种数据可视化”更明确，也更符合当前已有的 query index、diagnostics、mapping template、project package 等底座。

### 6.2 每份数据生成一张 Data Health Card

在导入完成后，为 source 或 recording 生成一张数据健康卡片：

- 数据类型、文件大小、行数或消息数。
- 时间范围、时间单位、是否单调、是否有时间缺口。
- 字段数量、空值比例、异常字段、疑似单位。
- 可视化入口、诊断入口、导出入口。
- 最高优先级 findings。

这张卡片可以成为 Dashboard、Recording 列表和诊断报告之间的共同摘要。

### 6.3 建立诊断配方 Recipe

把目前分散的模板能力升级为 recipe：

```text
Recipe = 数据类型识别 + Mapping 模板 + 可视化 Blueprint + 查询模板 + 诊断阈值 + 报告布局
```

示例：

- Robot Bag Health。
- Sensor CSV Monitor。
- CV Detection Review。
- Point Cloud Run Review。
- Experiment Comparison。

Recipe 可以先本地化，不急于做远程 marketplace。当前插件和模板 registry 已经能支撑这个方向。

### 6.4 让诊断结果可点击、可复现

每条 finding 应尽量包含：

- recording id。
- source id。
- entity path。
- topic 或字段名。
- time 或时间范围。
- 原始值和阈值。
- 建议动作。

用户点击后应跳到查询结果、样本预览或 Rerun 对应位置。这样诊断不只是静态报告，而是进入分析的导航系统。

## 7. 信息架构建议

| 一级区域 | 目标 | 关键功能 |
| --- | --- | --- |
| Dashboard | 快速理解当前项目状态 | 项目概览、最近 recording、健康卡片、批量任务、快速导入 |
| Import & Mapping | 把数据变成可用 recording | 选择源、schema profile、模板推荐、Mapping 编辑、校验、构建 |
| Visualize | 查看和回放数据 | 打开 Rerun、内联预览、artifact 状态、blueprint 管理 |
| Query & Compare | 快速筛选问题和比较运行 | 查询模板、无代码过滤、结果表、CSV/Parquet 导出、多 run 对比 |
| Diagnostics | 输出健康报告 | 诊断 preset、阈值、finding 过滤、报告导出、跳转定位 |
| Recipes & Extensions | 复用经验 | mapping template、visual template、diagnostic recipe、插件 |
| Settings | 本地运行配置 | 默认导出目录、artifact 目录、语言、任务并发、Rerun 选项 |

当前桌面端已有 Dashboard、Import、Recordings、Diagnostics、Templates、Settings，可以逐步把 Recordings 中的 Query/Compare 拆成更明确的 Query & Compare 区域，把 Visualize 能力从 Recording 列表中抽出来。

## 8. 核心体验原则

1. 先给结果，再给配置：导入后先自动生成 profile、Mapping、预览和诊断摘要，再允许用户深入编辑。
2. 每个问题都要有证据：finding 必须带时间、字段、值、阈值或 topic 信息。
3. 可视化和诊断互相跳转：图上看到的问题能生成查询，诊断报告里的问题能跳回图上。
4. 本地优先：默认离线运行，源数据不上传，项目包由用户主动导出。
5. 可复现：Mapping、模板、阈值、查询和报告都应保存在项目中。
6. 渐进复杂度：普通用户能一键导入和诊断，高级用户能编辑 Mapping、模板、decoder、校验模式和导出格式。

## 9. 未来发展路线

### 9.1 近期：补齐通用数据诊断

优先级最高。原因是它能把产品从机器人/CV 专用诊断扩展到更广泛的数据可视化和数据质量场景。

建议新增检查：

- 缺失值：全局缺失率、高缺失字段、连续缺失片段。
- 异常值：IQR、z-score、分位数阈值、突变点。
- 时间质量：重复时间戳、倒序、时间间隔过大、采样率估计、单位混用。
- 字段质量：常量列、低基数字段、高基数字段、类型混杂。
- 数据规模：行数、文件大小、消息量、帧数、topic 数。
- 分布漂移：同项目内多 recording 同字段分布差异。

输出形式复用现有 `DiagnosticReport`，新增 `data_quality`、`time_series_quality`、`schema_quality` 等 category。

### 9.2 近期：导入阶段的轻量内联图表

在 Mapping 确认前就给用户看到数据样子：

- scalar/time-series 折线预览。
- 数值字段直方图和箱线图。
- 状态字段频次条形图。
- 文本日志关键词摘要。
- 图片缩略图网格和检测框预览。
- 点云帧的轻量预览或统计摘要。

这些图表不需要替代 Rerun，而是降低用户进入 Rerun 前的判断成本。

### 9.3 中期：无代码查询与规则构建器

在当前内置 query template 之上增加：

- 字段选择器。
- 条件组合：大于、小于、等于、包含、正则、时间范围。
- 聚合：count、min、max、mean、latest、duration。
- 保存为查询模板。
- 查询结果一键导出或转为诊断规则。

这会让用户不改代码也能定义自己的“数据异常”。

### 9.4 中期：诊断与 Rerun 深度联动

建议实现：

- 从 finding 打开对应 Rerun recording。
- 自动定位 entity path。
- 尽量定位到时间点或时间范围。
- 在诊断报告中保存可复现的跳转上下文。
- 在 query result 中增加“查看上下文”动作。

### 9.5 中期：Recipe 系统

把现有 visual template、mapping template、query template、diagnostic preset 合并到更高层的 recipe。

一个 recipe 至少包含：

- 适用 source family。
- 字段匹配规则。
- 默认 semantic type 和 entity path。
- 默认 Rerun blueprint。
- 默认查询和诊断阈值。
- 报告展示布局。

先内置 4 个 recipe：Sensor CSV、Robot Bag、CV Detection、Point Cloud。后续允许用户从成功项目中导出 recipe。

### 9.6 长期：团队化与智能辅助

在本地优先原则不变的前提下，可以逐步发展：

- 远程 catalog 或共享项目索引。
- 项目版本与报告快照。
- 团队模板库。
- 诊断规则审阅和备注。
- 可选的 AI 辅助解释：总结报告、生成查询、推荐 Mapping 修复、解释异常原因。

AI 能力应作为可选层，不应成为基础导入、可视化和诊断流程的前置依赖。

## 10. 产品指标

| 指标 | 含义 |
| --- | --- |
| Time to First Preview | 从选择数据到看到 schema/预览的时间 |
| Time to First Recording | 从选择数据到生成 `.rrd` 的时间 |
| Import Success Rate | 导入并生成 Mapping 草稿的成功率 |
| Mapping Confirm Rate | 用户最终确认 Mapping 的比例 |
| Diagnostic Coverage | recording 中能生成有效 findings 或健康摘要的比例 |
| Query Latency | 常见查询模板返回首批结果的时间 |
| Reuse Rate | Mapping template 或 recipe 被复用的次数 |
| Export Rate | 查询、诊断或项目包导出的使用频率 |

## 12. 结论

当前项目已经具备成为数据可视化与快速诊断工具的核心基础：多源接入、自动 Mapping、Rerun 输出、查询索引、离线诊断、项目包和桌面/API/CLI 三端入口。下一步最值得投入的是通用数据诊断、轻量内联图表、无代码查询、诊断结果到可视化的跳转，以及 recipe 化模板体系。

如果产品目标是覆盖更多行业数据，不建议只堆更多图表类型。更好的方向是让 DataScope Studio 成为“从原始数据到可视化、从可视化到诊断、从诊断到可复现报告”的完整闭环。
