# DataScope Studio 软件设计报告

**基于 Rerun 的多模态工程数据可视化工作台**
版本：V1.0 设计稿
日期：2026-06-02

---

## 1. 项目定位

### 1.1 一句话定义

**DataScope Studio 是一个基于 Rerun 的本地优先、多模态、低代码数据可视化工作台。**

它让用户可以把 CSV、Parquet、JSONL、日志、图片、视频、点云、MCAP、ROS2 bag、检测结果、轨迹、传感器数据等导入进来，自动识别数据结构，自动映射到 Rerun 的 entity/component/archetype 模型，再通过可复用的 Blueprint 模板打开一个合适的可视化布局。

### 1.2 核心价值

Rerun 已经提供了强大的数据记录、Viewer、`.rrd` 文件、MCAP 处理、Blueprint 布局、dataframe 查询和 catalog server 能力；例如官方文档说明，Rerun 可以记录任意传感器数据、在 Viewer 中可视化、用 dataframe 查询，并支持跨一个或多个 recording 的工作流。([Rerun][1]) Rerun Viewer 可以直接打开 `.rrd`、`.mcap` 等文件，也可以通过 catalog server 组织多个 recording。([Rerun][1])

但是，Rerun 更偏 SDK / 数据层 / Viewer，而不是完整的“数据导入与可视化软件”。DataScope Studio 的价值是补齐这几层：

1. **无代码导入**：拖入文件，不写 Python 也能可视化。
2. **自动识别与映射**：自动判断哪些列是时间、坐标、图像、标签、状态、日志、检测框。
3. **模板化布局**：用 Robotics、CV、Sensor、Experiment、Geospatial 等模板一键生成 Rerun Blueprint。
4. **项目管理**：管理数据源、转换任务、`.rrd`、`.rbl`、实验 run、标签和参数。
5. **批量转换与对比**：把多个实验转换成统一 schema，进行并排对比和 dataframe 查询。
6. **插件化扩展**：用户可以添加自定义 adapter、transform、template。

### 1.3 不做什么

DataScope Studio **不是** Tableau / PowerBI / Excel 的替代品。它的核心场景不是业务报表，而是工程数据、机器人数据、CV 数据、传感器数据、仿真数据、嵌入式日志和实验数据。

它也不重新实现一个 2D/3D Viewer，而是把 Rerun Viewer 作为主要可视化引擎。Rerun 的数据模型基于 ECS：entity 表示被记录的对象，component 表示对象上的数据，archetype 是常见可视化对象的组件集合，例如 2D 点、3D 框等。([Rerun][2])

## 2. 目标用户与典型场景

### 2.1 目标用户

| 用户类型            | 典型需求                                      |
| --------------- | ----------------------------------------- |
| 机器人研发工程师        | 查看 ROS2 / MCAP、TF、点云、相机图像、轨迹、关节状态、诊断日志    |
| 计算机视觉工程师        | 查看图片/视频、检测框、mask、keypoints、置信度曲线、模型输出对比   |
| 自动驾驶 / SLAM 工程师 | 查看地图、点云、轨迹、pose graph、传感器时间同步             |
| 嵌入式 / IoT 工程师   | 查看多通道传感器波形、状态切换、异常点、设备日志                  |
| ML / 数据工程师      | 把多模态 recording 查询成 dataframe，做筛选、采样、训练集构建 |
| 测试与实验工程师        | 批量管理 run，标记失败片段，对比不同版本模型或固件               |

### 2.2 典型场景

#### 场景 A：CSV 传感器日志快速可视化

用户拖入：

```text
timestamp, imu_ax, imu_ay, imu_az, battery, state, error_code
```

DataScope Studio 自动识别：

```text
timestamp -> 时间轴
imu_ax/imu_ay/imu_az -> 多通道时间序列
battery -> scalar 曲线
state -> 状态时间线
error_code -> 日志 / 事件标记
```

然后生成：

```text
recording: sensor_run_001.rrd
blueprint: sensor_monitor.rbl
```

用户点击 **Open in Rerun**，看到曲线、状态、事件日志和时间轴联动。

#### 场景 B：ROS2 / MCAP 机器人调试

用户拖入 `.mcap` 文件。Rerun 已经支持 MCAP 文件处理，并能把 MCAP topic 映射到 Rerun entity，消息内容映射到 component，`message_log_time` 和 `message_publish_time` 也会作为不同 timeline 带入。([Rerun][3]) Rerun 的 MCAP 处理还包含 raw、schema、stats、protobuf、ros2msg、ros2_reflection、urdf 等层。([Rerun][3])

DataScope Studio 在这个基础上额外做：

```text
/topic 自动分类
TF / robot_description 自动归入机器人模板
相机 / 点云 / odom / joint_states 自动布置到 Robotics Debug Blueprint
异常 topic 自动标记
生成项目索引和实验元数据
```

#### 场景 C：CV 检测结果可视化

用户导入：

```text
images/
annotations.json
predictions.json
metrics.csv
```

系统生成：

```text
/camera/front/image
/camera/front/gt/boxes
/camera/front/pred/boxes
/camera/front/pred/scores
/metrics/precision
/metrics/recall
/logs/inference
```

Rerun 中打开后，可以同时看图像、框、mask、置信度、指标曲线和推理日志。

## 3. 产品总体设计

### 3.1 产品形态

推荐做成三种入口：

| 入口          | 说明                                | 优先级 |
| ----------- | --------------------------------- | --- |
| Desktop App | 主产品，拖拽导入、映射、模板、项目管理、打开 Rerun      | P0  |
| CLI         | 适合 CI、批处理、远程服务器、自动转换              | P0  |
| Python SDK  | 让高级用户写 adapter、transform、template | P1  |
| Team Server | 多人共享数据集、远程 catalog、权限、协作          | P2  |

### 3.2 核心工作流

```text
导入数据
  -> 文件/数据源检查
  -> 自动 schema 推断
  -> 用户确认映射
  -> 生成 Rerun recording
  -> 生成或选择 Blueprint
  -> 注册到本地项目 / catalog
  -> 打开 Rerun Viewer
  -> 查询、筛选、对比、导出
```

### 3.3 关键概念

| 概念        | 含义                                                   |
| --------- | ---------------------------------------------------- |
| Source    | 原始数据源，例如 CSV、MCAP、图片文件夹、JSONL、实时 socket              |
| Dataset   | 一组相关数据源的逻辑集合                                         |
| Stream    | 一个可视化数据流，例如 camera image、lidar points、battery scalar |
| Mapping   | 从原始字段到 Rerun entity path + archetype 的映射             |
| Recording | 生成后的 Rerun `.rrd` 文件                                 |
| Blueprint | Rerun 可视化布局 `.rbl` 或程序化 Blueprint                    |
| Template  | DataScope 的高级模板，负责生成 Blueprint 和推荐映射                 |
| Project   | 用户工作区，包含 source、dataset、recording、blueprint、metadata |
| Run       | 一次实验或一次设备采集，通常对应一个 recording 或多个 segment             |

## 4. Rerun 集成设计

### 4.1 Rerun 在系统中的角色

DataScope Studio 不替代 Rerun，而是围绕 Rerun 做产品化封装。

| 层级          | Rerun 负责                                   | DataScope Studio 负责          |
| ----------- | ------------------------------------------ | ---------------------------- |
| 数据记录        | `.rrd`、entity/component/archetype、timeline | 自动转换、统一 schema、字段映射          |
| 可视化         | Rerun Viewer、2D/3D/时间序列/日志/表格视图            | 模板选择、Blueprint 生成、Viewer 启动  |
| 布局          | Blueprint、`.rbl`、Viewer UI                 | 模板库、布局推荐、项目级保存               |
| 多 recording | catalog server、dataset、segment             | 项目管理、标签、run 对比、批量导入          |
| 查询          | dataframe query、DataFusion、Catalog SDK     | 查询 UI、预设分析、导出 Pandas/Parquet |
| MCAP        | MCAP importer / converter                  | topic 语义增强、机器人模板、异常检测        |

Rerun 的 Blueprint 明确区分了 recording 和显示方式：recording 提供数据，blueprint 决定这些数据如何展示；Blueprint 可以控制 Viewer 布局、panel、view 类型、颜色、时间范围等。([Rerun][4]) 这正适合 DataScope Studio 做“模板系统”。

### 4.2 Rerun 运行模式

#### 模式 1：离线文件模式

用于普通文件导入：

```text
raw data -> convert -> output.rrd -> rerun output.rrd
```

Rerun 官方支持通过命令打开 `.rrd` 文件：

```bash
rerun path/to/recording.rrd
```

并且 Viewer 可直接打开 `.rrd`、`.mcap` 等文件。([Rerun][1])

#### 模式 2：本地 catalog 模式

用于多 run 管理：

```text
多个 .rrd -> local rerun server -> dataset / segment -> viewer / query
```

Rerun 支持启动本地 catalog server，并将 `.rrd` 注册到 dataset 中，再通过 Viewer 浏览和查询。([Rerun][1]) 但需要注意，Rerun 文档说明开源 catalog server 会把数据加载到内存里，适合快速测试和本地实验；更大的持久化多人场景应设计自己的索引层，或接入 Rerun Hub / 其他后端。([Rerun][5])

#### 模式 3：实时流模式

用于实时设备、机器人、仿真：

```text
device / robot / simulator
  -> DataScope live adapter
  -> rr.RecordingStream
  -> Rerun Viewer
```

这种模式不一定先生成完整 `.rrd`，可以边采集边看，任务结束后再保存 recording。

#### 模式 4：MCAP 快速通道

对于 MCAP，优先调用 Rerun 原生能力：

```bash
rerun mcap convert input.mcap -o output.rrd
```

Rerun 官方文档给出了 MCAP 转 `.rrd` 的命令，并说明转换成 Rerun 原生格式可以更快加载。([Rerun][3]) DataScope Studio 在这条通道上增加 topic 分类、模板匹配、元数据索引和可视化配置。

#### 5.2.1 Desktop UI

建议技术栈：

```text
Tauri + React + TypeScript
```

主要界面：

| 页面                | 功能                                         |
| ----------------- | ------------------------------------------ |
| Project Home      | 项目列表、最近导入、最近打开的 recording                  |
| Import Wizard     | 拖拽导入、选择文件夹、选择数据源类型                         |
| Schema Inspector  | 显示文件字段、topic、采样率、时间范围、样例数据                 |

#### 5.2.2 Backend

建议第一版使用：

```text
Python backend + FastAPI + SQLite + Rerun Python SDK
```

原因：

1. Rerun Python SDK 适合快速生成 `.rrd`。
2. Python 生态适合读 CSV、Parquet、JSON、图片、视频、点云、ROS/MCAP。
3. 插件开发门槛低。

后续性能瓶颈出现后，可把 MCAP 大文件处理、点云转换、索引和压缩迁移到 Rust worker。

#### 5.2.3 Rerun Layer

核心组件：

| 组件                  | 职责                                       |
| ------------------- | ---------------------------------------- |
| RecordingWriter     | 把 canonical stream 写入 `.rrd`             |
| EntityPathBuilder   | 根据统一规范生成 Rerun entity path               |
| ArchetypeMapper     | 把 stream type 映射到 Rerun archetype        |
| BlueprintBuilder    | 生成 `.rbl` 或程序化 Blueprint                 |
| RerunProcessManager | 启动、关闭、连接 Rerun Viewer                    |
| CatalogManager      | 启动本地 `rerun server`、注册 `.rrd`            |
| QueryClient         | 通过 Catalog SDK / DataFusion 查询 dataframe |

Rerun 的 dataframe query 可在 Viewer 中交互使用，也可通过 Catalog SDK 程序化使用；它的目的就是把多速率、多 timeline、稀疏的机器人/传感器数据转换成 dataframe。([Rerun][6]) Rerun 查询 API 返回 DataFusion DataFrame，可继续过滤、投影、join，并可转 Pandas、Polars、PyArrow 等。([Rerun][7])

## 8. Adapter 设计

### 8.1 Adapter 生命周期

每个 adapter 必须实现四步：

```text
inspect -> infer_schema -> preview -> convert
```

| 阶段           | 说明                                        |
| ------------ | ----------------------------------------- |
| inspect      | 扫描文件结构、列名、topic、元数据、时间范围                  |
| infer_schema | 推断 semantic_type、time_key、单位、维度           |
| preview      | 返回可预览样本，例如前 100 行、缩略图、topic 列表            |
| convert      | 根据 mapping 写出 `.rrd` 或中间 canonical stream |

### 8.2 Adapter 接口

```python
from dataclasses import dataclass
from typing import Protocol, Any

@dataclass
class SourceInfo:
    source_id: str
    source_type: str
    path: str
    metadata: dict[str, Any]

@dataclass
class StreamInfo:
    stream_id: str
    name: str
    semantic_type: str
    fields: list[str]
    time_key: str | None
    confidence: float
    metadata: dict[str, Any]

@dataclass
class ConvertRequest:
    source: SourceInfo
    mappings: list[dict[str, Any]]
```

### 8.3 内置 Adapter

#### CSV / Parquet Adapter

识别规则：

| 规则                                   | 识别结果                        |
| ------------------------------------ | --------------------------- |
| `timestamp`, `time`, `t`, `datetime` | 时间列                         |
| 全数字列                                 | scalar / timeseries         |
| `x,y,z` 或 `pos_x,pos_y,pos_z`        | trajectory3d / points3d     |
| `lat,lon` / `latitude,longitude`     | geo_points                  |
| 字符串列                                 | state / text_log / category |

转换策略：

1. 小文件：直接读入 DataFrame。
2. 中大文件：Polars / PyArrow 分块读取。
3. 时间序列：优先使用 Rerun 的 column-oriented 写入方式。Rerun 文档说明，当数据已经是 column-oriented 时，`send_columns` 能一次发送多个 index/component column，比逐行 log 更合适。([Rerun][10])

## 9. 自动映射设计

### 9.1 映射目标

自动映射要把原始数据变成：

```text
原始字段 / topic / 文件
  -> semantic_type
  -> Rerun entity_path
  -> Rerun archetype
  -> timeline
  -> visualization template slot
```

### 9.2 映射评分模型

每个候选映射计算置信度：

```text
score =
  0.30 * name_match_score
+ 0.25 * dtype_score
+ 0.20 * shape_score
+ 0.15 * value_distribution_score
+ 0.10 * domain_rule_score
```

示例：

| 字段                    | 识别依据               | 输出                  |
| --------------------- | ------------------ | ------------------- |
| `timestamp`           | 名称匹配 + 单调递增        | time                |
| `battery`             | 名称匹配 + 0~1 或 0~100 | scalar              |
| `state`               | 字符串/枚举值少           | state               |
| `message`             | 字符串长文本             | text_log            |
| `x,y,z`               | 三列数字               | position / points3d |
| `qx,qy,qz,qw`         | 四元数模式              | rotation            |
| `xmin,ymin,xmax,ymax` | bbox 模式            | boxes2d             |
| `lat,lon`             | 经纬度范围              | geo_points          |

### 9.3 用户确认界面

Mapping Editor 采用三栏：

```text
左侧：原始字段 / topic / 文件树
中间：自动识别结果和置信度
右侧：Rerun entity path + archetype + template slot
```

交互：

```text
拖拽字段到 stream
合并多个字段为一个 stream
拆分 stream
改 entity path
改 archetype
保存 mapping recipe
```

## 10. Blueprint 与模板系统设计

### 10.1 为什么模板用 Blueprint 实现

Rerun Blueprint 控制 Viewer 的布局、panel 显示、view 类型、视觉属性等；Blueprint 可以在 Viewer UI 中交互修改，也可以保存为 `.rbl` 文件，或通过 API 程序化生成。([Rerun][4]) 官方文档还说明，相同 Application ID 的 recordings 会使用相同 blueprint，这使得“同类实验共享布局”非常自然。([Rerun][4])

```yaml
template:
  id: robotics_debug
  name: Robotics Debug
  version: 1.0.0
  app_id: datascope.robotics.v1

  required_streams:
    - semantic_type: points3d
      slot: point_cloud
      optional: true
    - semantic_type: image
      slot: camera_image
      optional: true
    - semantic_type: transform3d
      slot: tf_tree
      optional: true
    - semantic_type: scalar
      slot: metrics
      optional: true
    - semantic_type: text_log
      slot: diagnostics
      optional: true

  entity_roots:
    world: /world
    sensors: /sensors
    metrics: /metrics
    logs: /logs

  layout:
    type: grid
    views:
      - id: world_3d
        type: Spatial3DView
        contents:
          - /world/**
          - /sensors/lidar/**
      - id: camera
        type: Spatial2DView
        contents:
          - /sensors/camera/front/**
      - id: metrics
        type: TimeSeriesView
        contents:
          - /metrics/**
      - id: logs
        type: TextLogView
        contents:
          - /logs/**
```

### 10.4 模板匹配算法

```text
输入：streams[]
输出：推荐 templates[]

for each template:
  score = 0
  for each required_stream:
    if matching stream exists:
      score += required_stream.weight
    elif required_stream.optional:
      score += 0
    else:
      score -= penalty

返回 score 最高的前 N 个模板
```

示例：

```text
存在 image + boxes2d + scores -> 推荐 CV Detection
存在 points3d + transform3d + image + joint_states -> 推荐 Robotics Debug
存在 scalar 多列 + state + text_log -> 推荐 Sensor Monitor
存在 lat/lon + speed + altitude -> 推荐 Geospatial Trace
```

## 11. 转换与写入策略

### 11.1 离线数据转换流程

```text
Source Reader
  -> Chunk / Batch Reader
  -> Canonical Stream
  -> Mapping Resolver
  -> Rerun RecordingWriter
  -> .rrd
  -> Validation
  -> Blueprint
  -> Project Index
```

### 11.2 写入策略

| 数据类型   | 推荐写入方式                                  |
| ------ | --------------------------------------- |
| 小量实时数据 | `rr.log`                                |
| 大量时间序列 | `send_columns`                          |
| 图片/视频  | 编码图像或帧引用                                |
| 点云     | 分块 / 降采样预览 / 完整写入                       |
| MCAP   | 先用 Rerun MCAP importer / converter      |
| 原始表格   | dataframe / AnyValues / sidecar Parquet |
| 自定义字段  | `AnyValues` 或自定义 component              |

Rerun 的性能与 chunk 数量高度相关；官方文档说明，Rerun 把数据存储为 Arrow-encoded chunks，很多小 chunk 会带来更高开销，而较大的 chunk 能降低索引和 per-chunk 开销，提高写入与查询吞吐。([Rerun][12]) 所以 DataScope Studio 的转换器要避免逐行写大量数据，应优先批量写入。

### 11.3 大文件策略

| 问题           | 策略                                               |
| ------------ | ------------------------------------------------ |
| CSV 超大       | 分块读取，预览只采样，完整转换后台执行                              |
| 点云过大         | 预览降采样，完整数据保留；允许按时间片导出                            |
| 视频过大         | 默认引用视频或抽关键帧，不强制展开所有帧                             |
| MCAP 过大      | 优先 Rerun 原生 MCAP reader / converter；按 topic 过滤   |
| 多 run 对比     | 统一 app_id、entity path、timeline；必要时生成对比 recording |
| catalog 内存限制 | MVP 本地小规模；大规模转团队后端或外部对象存储                        |

### 11.4 Chunk Processing API 的使用边界

Rerun 目前有 Chunk Processing API，面向 chunk-centric 的数据导入、转换和 pipeline，可处理常见机器人文件格式，并提供多线程 native pipeline 执行；但官方也标注该 API 当前是 experimental，未来可能变化。([Rerun][13]) 因此：

```text
V1 稳定路径：Rerun SDK + MCAP converter + send_columns
V1 实验路径：可选使用 rerun.experimental 的 Chunk Processing API
V2 之后：如果 API 稳定，再深度集成为高性能转换引擎
```

### 18.1 性能目标

| 场景                 | 目标                 |
| ------------------ | ------------------ |
| 100MB CSV inspect  | 5 秒内生成初步 schema    |
| 1GB CSV conversion | 后台任务，不阻塞 UI        |

| 多 run 项目           | 支持按标签、时间、参数快速查找    |
| Viewer 打开          | 优先使用 `.rrd`，避免重复转换 |

### 18.2 缓存策略

```text
schema cache
preview cache
thumbnail cache
sampled parquet cache
mapping cache
template match cache
conversion artifact cache
```

### 18.3 错误恢复

每个转换 job 保存 checkpoint：

```yaml
job:
  id: job_001
  source: source_001
  current_stage: convert_streams
  completed_streams:
    - battery
    - imu
  failed_streams:
    - camera
  can_resume: true
```

如果某个 stream 转换失败，系统不应整体失败，而应：

```text
保留成功 stream
标记失败 stream
生成 warning
允许用户修正 mapping 后重试
```

### 18.4 数据校验

转换后自动检查：

```text

```

## 23. 风险与应对

| 风险           | 影响                   | 应对                                              |
| ------------ | -------------------- | ----------------------------------------------- |
| Rerun API 变化 | 转换层和 Blueprint 层需要调整 | 固定版本，封装 RerunLayer，做兼容测试                        |
| 大文件内存压力      | UI 卡顿、转换失败           | 分块读取、预览采样、后台 job、topic/filter                   |
| 自动映射错误       | 可视化结果不符合预期           | 显示置信度，必须用户确认关键映射                                |
| 插件不安全        | 文件破坏或数据泄露            | 插件 manifest、权限控制、venv、子进程隔离                     |
| MCAP 自定义消息复杂 | 无法直接语义可视化            | 利用 schema reflection + 用户 mapping + custom lens |
| Viewer 嵌入限制  | UI 一体化体验不够           | MVP 外部启动 Rerun，后续再探索嵌入                          |
| 团队多用户复杂      | 权限和存储成本高             | V1 本地优先，P2 再做 server/team mode                  |

## 25. 最终架构结论

DataScope Studio 应该采用“**DataScope 做工程化入口，Rerun 做可视化与数据层**”的架构。

最核心的设计原则是：

1. **不要重写 Viewer**：Rerun Viewer 已经覆盖空间、时间序列、日志、dataframe、timeline 等核心能力。
2. **统一 schema 比支持更多格式更重要**：只要 mapping 稳定，模板、对比、查询才能复用。
3. **Blueprint 是产品壁垒**：把复杂可视化经验沉淀成模板，而不是让用户每次手调布局。
4. **Adapter 插件化**：数据格式无穷多，核心团队只维护主流格式。
5. **本地优先**：工程数据通常大且敏感，MVP 必须默认本地可用。
6. **批量与实时都支持，但分阶段做**：先离线文件，后实时流。
7. **Rerun 版本隔离**：所有 Rerun API 调用集中在 RerunLayer，避免全项目被 API 变化影响。

最小可行产品可以压缩成一句话：

> **一个可以拖入数据、自动映射、选择模板、生成 `.rrd + .rbl` 并一键打开 Rerun 的工程数据可视化工作台。**
